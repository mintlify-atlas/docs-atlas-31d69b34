## Overview

The Roles API provides endpoints for managing roles and permissions in the SMyEG platform. This includes creating custom roles, assigning permissions, and managing role-based access control (RBAC) for users.

## Authentication

All Roles API endpoints require authentication via session token. Include the session token in your request headers:

```http
Authorization: Bearer <your-session-token>
```

## Authorization

Roles API endpoints require specific permissions:
- **SUPER_ADMIN** or **ADMIN** roles have full access to role management
- Other users require specific module permissions: `users:READ`, `users:CREATE`, `users:UPDATE`, `users:DELETE`
- **SUPER_ADMIN** can manage roles across all organizations
- **ADMIN** can only manage roles within their organization
- System roles (built-in roles) cannot be deleted or have their slugs modified

## Create Role

<RequestExample>
```bash cURL
curl -X POST "https://api.smyeg.com/api/roles" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Project Manager",
    "slug": "PROJECT_MANAGER",
    "description": "Manages forest projects and field operations"
  }'
```

```javascript JavaScript
const response = await fetch('https://api.smyeg.com/api/roles', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer <token>',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: 'Project Manager',
    slug: 'PROJECT_MANAGER',
    description: 'Manages forest projects and field operations'
  })
});
const data = await response.json();
```
</RequestExample>

<ResponseExample>
```json Response

  "data": {

</ResponseExample>

### Request Body

<ParamField body="name" type="string" required>
  Human-readable role name
</ParamField>

<ParamField body="slug" type="string" required>
  Role slug (will be normalized to uppercase with underscores)
</ParamField>

<ParamField body="description" type="string">
  Role description
</ParamField>

### Notes

- Slug is automatically normalized: trimmed, uppercased, spaces/special characters replaced with underscores
- Role is created for the user's organization (derived from session)
- Role slug must be unique within the organization
- An audit log entry is automatically created
- New roles have no permissions by default - use the Update Role Permissions endpoint to assign permissions

### Request Body

<ParamField body="roleId" type="string" required>
  Role UUID to update permissions for
</ParamField>

<ParamField body="permissionIds" type="array" required>
  Array of permission UUIDs to assign to the role. Pass an empty array to remove all permissions.
</ParamField>

### Notes

- This endpoint **replaces** all existing permissions with the provided list
- To remove all permissions, pass an empty array: `permissionIds: []`
- Invalid permission IDs are filtered out automatically
- Duplicate permission IDs are removed
- Use the GET /api/roles endpoint to retrieve available permission IDs from the modules list

  }

```
</ResponseExample>

### Path Parameters

<ParamField path="id" type="string" required>
  Role UUID
</ParamField>

### Notes

- System roles (isSystemRole: true) **cannot be deleted**
- This performs a **soft delete** - the role is marked as inactive (isActive: false)
- All active user role assignments are deactivated
- An audit log entry is automatically created
- Users with this role will lose access to associated permissions immediately

## Permission Actions

The RBAC system supports the following permission actions:

| Action | Description |
|--------|-------------|
| `CREATE` | Create new resources |
| `READ` | Read/view resources |
| `UPDATE` | Update existing resources |
| `DELETE` | Delete resources |
| `EXPORT` | Export data |
| `ADMIN` | Full administrative access to the module |

---

## System Roles

The following system roles are built-in and cannot be deleted:

### SUPER_ADMIN
Full system access across all organizations and modules. Permissions:
- `dashboard:READ`
- `users:ADMIN`
- `organizations:ADMIN`
- `forest-patrimony:ADMIN`
- `forest-biological-asset:ADMIN`
- `asset-measurement:ADMIN`
- `forest-config:ADMIN`
- `general-config:ADMIN`
- `profile:ADMIN`
- `analytics:ADMIN`
- `settings:ADMIN`
- `audit:ADMIN`

### ADMIN
Organization administrator with full access within their organization. Permissions:
- `dashboard:READ`
- `forest-patrimony:READ`, `CREATE`, `UPDATE`, `DELETE`
- `forest-biological-asset:READ`
- `asset-measurement:READ`, `CREATE`, `UPDATE`
- `settings:READ`, `UPDATE`
- `users:READ`, `CREATE`, `UPDATE`, `DELETE`

### GERENTE_CAMPO (Field Manager)
Field operations manager. Permissions:
- `forest-patrimony:READ`
- `forest-biological-asset:READ`
- `asset-measurement:READ`
- `users:READ`

### CONTADOR (Accountant)
Accounting and financial reporting. Permissions:
- `forest-patrimony:READ`

### USER
Basic user access. Permissions:
- `dashboard:READ`

## RBAC Permission Check

When checking permissions in the API:

```javascript
// Permission format: "module:action"
const hasPermission = requirePermission(
  user.permissions,
  "forest-patrimony",
  "CREATE"
);
```

Permissions are checked in the following order:
1. **SUPER_ADMIN** and **ADMIN** roles bypass permission checks
2. Permission format: `{moduleSlug}:{action}`
3. The user's effective permissions are derived from their active roles

## Error Responses

All endpoints return standard error responses:

```json
{
  "success": false,
  "error": {
    "message": "Error message",
    "code": 400,
    "details": {}
  }
}
```

### Common Error Codes

- `400` - Bad Request (invalid parameters or request body)
- `401` - Unauthorized (missing or invalid authentication)
- `403` - Forbidden (insufficient permissions)
- `404` - Not Found (role not found)
- `409` - Conflict (role slug already exists)
- `500` - Internal Server Error

## Best Practices

1. **Use descriptive role names**: Choose clear, meaningful names that reflect the role's purpose
2. **Follow naming conventions**: Use uppercase with underscores for slugs (e.g., `PROJECT_MANAGER`)
3. **Principle of least privilege**: Only assign permissions that are necessary for the role
4. **Regular audits**: Periodically review roles and permissions to ensure they're still appropriate
5. **Document custom roles**: Maintain documentation of custom roles and their intended purpose
6. **Test permission changes**: Verify that permission changes work as expected before deploying to production
