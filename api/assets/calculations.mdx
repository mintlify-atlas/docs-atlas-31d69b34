# Asset Calculation Jobs API

Trigger and manage automated calculation jobs that process measurement observations and generate statistical summaries and metrics.

## Overview

Calculation jobs run asynchronously to analyze campaign observations using protocol-specific formulas. They generate result snapshots with computed metrics, quality indicators, and decision support classifications.

### Job Lifecycle

1. **PENDING** - Job created, waiting to run
2. **PROCESSING** - Job is currently executing
3. **COMPLETED** - Job finished successfully
4. **FAILED** - Job encountered an error

### Worker Integration

Jobs are processed by the asset measurement worker (`src/lib/asset-measurement-worker.ts`) which:
- Fetches all observations for the campaign
- Applies protocol-specific calculation formulas
- Generates quality metrics and validation reports
- Creates versioned result snapshots
- Handles errors and retry logic

## Create Calculation Job

<api-endpoint method="POST" path="/api/assets/calculation-jobs">
Enqueue a new calculation job for a campaign.
</api-endpoint>

### Request Body

<ParamField body="campaignId" type="uuid" required>
  Campaign ID to process
</ParamField>

<ParamField body="runAfter" type="datetime">
  Schedule job to run after this time (defaults to now)
</ParamField>

<ParamField body="payload" type="object">
  Custom job parameters (JSON object)
</ParamField>

### Response

Returns the created job object with HTTP status `201 Created`. The job is automatically enqueued and will be processed by the worker.

<Note>
  The campaign must exist and belong to the user's organization. Domain and protocol type are inherited from the campaign.
</Note>

### Example Request

```bash
curl -X POST "https://api.example.com/api/assets/calculation-jobs" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "campaignId": "campaign-uuid",
    "runAfter": "2026-03-16T00:00:00Z"
  }'
```

### Example Response

```json
{

  "protocolType": "BIO_SOBREVIVENCIA",
```

## Calculation Worker Logic

The asset measurement worker (`processAssetCalculationJob`) performs the following steps:

### 1. Fetch Campaign Data

Retrieves the campaign with all observations:
```typescript
const job = await prisma.assetCalculationJob.findUnique({
  where: { id: jobId },
  include: {
    campaign: {
      select: {
        observations: {
          select: {
            sectionKey: true,
            level5UnitCode: true,
            observedAt: true,
            payload: true,
          },
        },
      },
    },
  },
});
```

### 2. Calculate Summary

Generates observation counts by section and Level 5 unit:
```typescript
{
  observationCount: 45,
  sectionCount: 3,
  level5UnitCount: 12,
  observationsBySection: { "L1": 15, "L2": 15, "L3": 15 },
  observationsByLevel5Unit: { "P001": 4, "P002": 3, ... }
}
```

### 3. Apply Protocol Formulas

Runs specialized calculation functions based on protocol type:

**BIO_SOBREVIVENCIA** calculates:
- `SURVIVAL_REAL_PCT` - Actual survival percentage
- `SURVIVAL_PRESC_PCT` - Prescribed survival percentage
- `DENSITY_VIVA_HA` - Live density per hectare
- `AVG_HEIGHT_RODAL` - Average stand height
- `SAMPLING_ERROR_PCT` - Statistical sampling error
- `SURVIVAL_SEMAFORO` - Status indicator (VERDE/AMARILLO/ROJO/GRIS)
- Decision classifications for acceptance and next steps

**HIDRO_ESTACION_HIDROLOGICA** calculates:
- `MEAN_FLOW_M3S` - Mean flow rate
- `MAX_FLOW_M3S` - Maximum flow
- `MEAN_WATER_LEVEL_M` - Mean water level
- `FLOW_VARIABILITY_PCT` - Flow variability
- `HYDRO_ALERT_DAYS` - Number of alert days
- `HYDRO_STATUS` - Risk classification

**DENDRO_PARCELA_CARBONO** calculates:
- `BIOMASS_T_HA` - Biomass per hectare
- `CARBON_STOCK_T_HA` - Carbon stock (biomass × 0.47)
- `CO2_EQ_T_HA` - CO2 equivalent (carbon × 44/12)
- `CARBON_CLASS` - Capture classification

### 4. Create Result Snapshot

Stores results with versioning:
```typescript
await tx.assetResultSnapshot.create({
  data: {
    campaignId: job.campaignId,
    level4Id: job.campaign.level4Id,
    domain: job.campaign.domain,
    resultKey: `${campaignCode}_AUTO_SUMMARY`,
    resultVersion: nextVersion,
    schemaVersion: "1.0",
    resultPayload: {
      summary,
      metrics,
      quality,
      protocolType,
      generatedBy: "asset-measurement-worker",
      formulaVersion: "1.0.0",
    },
    isCurrent: true,
    calculatedAt: new Date(),
  },
});
```

### 5. Quality Metrics

All protocols include quality indicators:
- `validObservationCount` - Observations passing validation
- `invalidObservationCount` - Observations with errors
- `totalObservationCount` - Total observations processed
- Protocol-specific quality flags

## Worker Scheduler

The worker scheduler (`src/workers/asset-measurement-worker-scheduler.ts`) runs periodically to process pending jobs:

```typescript
// Finds next pending job where runAfter <= now
const nextJob = await prisma.assetCalculationJob.findFirst({
  where: {
    status: AssetCalculationStatus.PENDING,
    runAfter: { lte: new Date() },
  },
  orderBy: { createdAt: 'asc' },
});

if (nextJob) {
  await processAssetCalculationJob(nextJob.id);
}
```

## Permissions

All calculation job endpoints require authentication and appropriate permissions:

- **GET**: Requires `asset-measurement:READ` or any write permission
- **POST**: Requires `asset-measurement:CREATE`
- **DELETE**: Requires `asset-measurement:DELETE`

Super admins bypass organization restrictions.

## Audit Logging

All create and delete operations are logged to the audit log with:
- User ID
- Action type (CREATE, DELETE)
- Entity type: `AssetCalculationJob`
- Entity ID
- Changed values

## Error Handling

Jobs that fail during processing:
- Set status to `FAILED`
- Store error message in `errorMessage` field
- Increment `attempts` counter
- Do not automatically retry (manual intervention required)

<Note>
  For production deployments, consider implementing a retry mechanism with exponential backoff for transient failures.
</Note>
