## Overview

The Users API provides endpoints for managing users in the SMyEG platform. This includes creating, listing, updating, and deleting users, as well as managing their roles and organization assignments.

## Authentication

All Users API endpoints require authentication via session token. Include the session token in your request headers:

```http
Authorization: Bearer <your-session-token>
```

## Authorization

Users API endpoints require specific permissions:
- **ADMIN** or **SUPER_ADMIN** roles have full access
- Other users require specific module permissions: `users:READ`, `users:CREATE`, `users:UPDATE`, `users:DELETE`, `users:EXPORT`
- **SUPER_ADMIN** can manage users across all organizations
- **ADMIN** can only manage users within their organization

## Create User

<RequestExample>
```bash cURL
curl -X POST "https://api.smyeg.com/api/users" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "newuser@example.com",
    "firstName": "Jane",
    "lastName": "Smith",
    "roleSlug": "USER",
    "password": "SecurePass123!",
    "organizationId": "org-uuid"
  }'
```

```javascript JavaScript
const response = await fetch('https://api.smyeg.com/api/users', {
  method: 'POST',

    'Content-Type': 'application/json'
  },

    firstName: 'Jane',
    lastName: 'Smith',
    roleSlug: 'USER',
    password: 'SecurePass123!',
    organizationId: 'org-uuid'

});
```

### Request Body

<ParamField body="email" type="string" required>
  User's email address (must be unique)
</ParamField>

<ParamField body="firstName" type="string">
  User's first name (minimum 2 characters)
</ParamField>

<ParamField body="lastName" type="string">
  User's last name (minimum 2 characters)
</ParamField>

<ParamField body="roleSlug" type="string" required>
  Role slug to assign to the user (e.g., `ADMIN`, `USER`, `CONTADOR`, `GERENTE_CAMPO`)
</ParamField>

<ParamField body="password" type="string">
  User's password (minimum 8 characters). If not provided, a random password is generated and user status is set to `PENDING_VERIFICATION`
</ParamField>

<ParamField body="organizationId" type="string" required>
  Organization UUID to assign the user to
</ParamField>

### Notes

- If `password` is provided, user status is set to `ACTIVE`
- If `password` is omitted, user status is set to `PENDING_VERIFICATION` and a random password is generated
- An audit log entry is automatically created for user creation
- The role must exist or will be provisioned automatically based on the roleSlug

## Delete User

<RequestExample>
```bash cURL
curl -X DELETE "https://api.smyeg.com/api/users/{id}" \
  -H "Authorization: Bearer <token>"
```

```javascript JavaScript
const response = await fetch('https://api.smyeg.com/api/users/{id}', {
  method: 'DELETE',
  headers: {
    'Authorization': 'Bearer <token>'
  }
});
const data = await response.json();
```
</RequestExample>

<ResponseExample>
```json Response
{
  "success": true,
  "data": {
    "message": "Usuario eliminado lógicamente"
  }
}
```
</ResponseExample>

### Path Parameters

<ParamField path="id" type="string" required>
  User UUID
</ParamField>

### Notes

- This performs a **soft delete** - the user is not permanently removed
- User status is set to `DELETED`
- User email is obfuscated: `deleted_{timestamp}_{id}@invalid.local`
- All active sessions are revoked
- An audit log entry is automatically created
- The user cannot log in after deletion

### Request

<ParamField body="file" type="file" required>
  CSV or XLSX file containing user data

### File Format

Supported file formats: **CSV** and **XLSX**

**Required columns:**
- `email` - User's email address
- `roleSlug` / `role` / `rol` - Role to assign

**Optional columns:**
- `firstName` / `nombres` / `nombre` - User's first name
- `lastName` / `apellidos` / `apellido` - User's last name
- `password` / `contrasena` / `contraseña` - User's password
- `organizationId` / `organization` / `organizacionId` / `organizacion` - Organization reference (ID, name, or slug)

**Example CSV:**
```csv
email,firstName,lastName,roleSlug,password,organizationId
user1@example.com,John,Doe,USER,Pass123!,org-uuid
user2@example.com,Jane,Smith,ADMIN,,my-organization
```

### Response Fields

<ResponseField name="created" type="integer">
  Number of new users created
</ResponseField>

<ResponseField name="updated" type="integer">
  Number of existing users updated
</ResponseField>

<ResponseField name="skipped" type="integer">
  Number of rows skipped due to errors
</ResponseField>

<ResponseField name="errors" type="array">
  Array of error objects for failed rows
  
  <ResponseField name="row" type="integer">
    Row number in the file (1-based, excluding header)
  </ResponseField>
  
  <ResponseField name="email" type="string">
    Email from the failed row
  </ResponseField>
  
  <ResponseField name="error" type="string">
    Error message describing the failure
  </ResponseField>
</ResponseField>

### Notes

- If a user with the same email exists, they will be updated (if permissions allow)
- Organization can be referenced by UUID, name, or slug
- Non-SUPER_ADMIN users can only import users to their own organization
- If password is not provided, a random password is generated
- Column names are case-insensitive and accent-insensitive

### Export Format

**Exported columns:**
- `email` - User's email address
- `firstName` / `Nombres` - User's first name
- `lastName` / `Apellidos` - User's last name
- `status` / `Estatus` - User status
- `roleSlug` / `Rol` - Assigned role
- `organizationId` / `Organización ID` - Organization UUID
- `organizationName` / `Organización` - Organization name
- `createdAt` / `Fecha` - Creation timestamp

### Notes

- Maximum export limit is configurable via `USERS_EXPORT_MAX_LIMIT` environment variable (default: 5000)
- Files are named with the current date: `users_2024-01-15.csv` or `users_2024-01-15.xlsx`
- CSV files use UTF-8 encoding
- XLSX files include proper formatting and headers

## Error Responses

All endpoints return standard error responses:

```json
{
  "success": false,
  "error": {
    "message": "Error message",
    "code": 400,
    "details": {}
  }
}
```

### Common Error Codes

- `400` - Bad Request (invalid parameters or request body)
- `401` - Unauthorized (missing or invalid authentication)
- `403` - Forbidden (insufficient permissions)
- `404` - Not Found (user not found)
- `409` - Conflict (user already exists)
- `500` - Internal Server Error

## User Status Values

| Status | Description |
|--------|-------------|
| `ACTIVE` | User account is active and can log in |
| `INACTIVE` | User account is inactive |
| `PENDING_VERIFICATION` | User awaiting email verification |
| `LOCKED` | Account locked due to failed login attempts or security |
| `DELETED` | Soft-deleted account (cannot log in) |

---

## Available Roles

| Role Slug | Description |
|-----------|-------------|
| `SUPER_ADMIN` | Full system access across all organizations |
| `ADMIN` | Organization administrator with full access |
| `CONTADOR` | Accountant role with read access to forest patrimony |
| `GERENTE_CAMPO` | Field manager with read access to users and forest data |
| `USER` | Basic user with dashboard access |

Custom roles can be created via the [Roles API](/api/roles/overview).
