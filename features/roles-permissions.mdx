## Overview

SMyEG implements a **comprehensive Role-Based Access Control (RBAC)** system with:

- Module-based permission structure
- Action-level granularity (CREATE, READ, UPDATE, DELETE, ADMIN)
- CASL integration for declarative authorization
- Organization-scoped and system-wide roles

## Permission Structure

Permissions follow a **module:action** format:

```
<moduleSlug>:<action>
```

Examples:
- `users:READ`
- `forest-patrimony:CREATE`
- `organizations:ADMIN`

### Permission Actions

<ParamField path="CREATE" type="action">
  Allows creating new records in the module
</ParamField>

<ParamField path="READ" type="action">
  Allows viewing/listing records in the module
</ParamField>

<ParamField path="UPDATE" type="action">
  Allows modifying existing records
</ParamField>

<ParamField path="DELETE" type="action">
  Allows removing records
</ParamField>

<ParamField path="ADMIN" type="action">
  Grants full access to the module (all actions)
</ParamField>

<ParamField path="EXPORT" type="action">
  Allows exporting data from the module
</ParamField>

## Fetching User Permissions

The `getUserRolesAndPermissions` function loads a user's complete permission set:

```typescript src/lib/permissions.ts
export async function getUserRolesAndPermissions(userId: string) {
  const roleAssignments = await prisma.userRole.findMany({
    where: { userId, isActive: true },
    include: {
      role: {
        include: {
          rolePermissions: {
            include: {
              permission: {
                include: { module: true }
              }
            }
          }
        }
      }
    }
  });

  const roles = roleAssignments.map((assignment) => assignment.role.slug);
  const permissions = roleAssignments.flatMap((assignment) =>
    assignment.role.rolePermissions.map(
      (rp) => `${rp.permission.module.slug}:${rp.permission.action}`
    )
  );

  return {
    roles,
    permissions: Array.from(new Set(permissions))
  };
}
```

<Note>
  Permissions from multiple role assignments are automatically merged and deduplicated.
</Note>

## Permission Checking

### Server-Side Check

Use `hasPermission` for simple permission checks:

```typescript src/lib/permissions.ts
export function hasPermission(
  permissions: string[], 
  moduleSlug: string, 
  action: string
) {
  return (
    permissions.includes(`${moduleSlug}:${action}`) || 
    permissions.includes(`${moduleSlug}:ADMIN`)
  );
}
```

Example usage:

```typescript
const session = await auth();
const { permissions } = await getUserRolesAndPermissions(session.user.id);

if (!hasPermission(permissions, "users", "CREATE")) {
  return { error: "Unauthorized" };
}
```

### CASL Integration

For complex authorization logic, use CASL abilities:

```typescript src/lib/ability.ts
import { AbilityBuilder, AbilityClass, PureAbility } from "@casl/ability";

export type AppAction = "manage" | "create" | "read" | "update" | "delete";
export type AppSubject = string;
export type AppAbility = PureAbility<[AppAction, AppSubject]>;

const actionMap: Record<string, AppAction> = {
  CREATE: "create",
  READ: "read",
  UPDATE: "update",
  DELETE: "delete",
  ADMIN: "manage"
};

export function buildAbilityFromPermissions(permissions: string[]) {
  const { can, build } = new AbilityBuilder<AppAbility>(
    PureAbility as AbilityClass<AppAbility>
  );

  for (const entry of permissions) {
    const [moduleSlug, actionRaw] = entry.split(":");
    const action = actionMap[actionRaw] ?? "read";
    if (moduleSlug) {
      can(action, moduleSlug);
    }
  }

  return build();
}
```

#### Using CASL Abilities

```typescript src/app/(dashboard)/organizaciones/page.tsx
const session = await auth();
if (!session?.user?.id) {
  redirect("/login");
}

const livePermissionInfo = await getUserRolesAndPermissions(session.user.id);
const ability = buildAbilityFromPermissions(
  livePermissionInfo?.permissions ?? []
);

if (!ability.can("read", "organizations")) {
  redirect("/unauthorized");
}

// User is authorized to read organizations
```

## Helper Functions

### Access Level Checks

```typescript src/lib/rbac.ts
export function canAccessOrganizations(role: AppRole | null) {
  return role === "ADMIN" || role === "SUPER_ADMIN";
}

export function canAccessUsers(role: AppRole | null) {
  return role === "ADMIN" || 
         role === "SUPER_ADMIN" || 
         role === "GERENTE_CAMPO";
}
```

### Role Normalization

```typescript src/lib/rbac.ts
export function normalizeRole(value: string | null | undefined): AppRole | null {
  if (!value) return null;
  const normalized = value.trim().toUpperCase();
  return APP_ROLES.includes(normalized as AppRole) 
    ? (normalized as AppRole) 
    : null;
}
```

### Email-Based Role Inference

```typescript src/lib/rbac.ts
export function inferRoleFromEmail(email: string): AppRole {
  const normalized = email.trim().toLowerCase();
  if (normalized.includes("admin")) return "ADMIN";
  if (normalized.includes("gerente")) return "GERENTE_CAMPO";
  if (normalized.includes("contador")) return "CONTADOR";
  return "USER";
}
```

## Best Practices

<AccordionGroup>
  <Accordion title="Check Permissions on Every Request">
    Always verify permissions at the API route or page level, never rely solely on UI hiding:
    
    ```typescript
    const session = await auth();
    const { permissions } = await getUserRolesAndPermissions(session.user.id);
    
    if (!hasPermission(permissions, "users", "DELETE")) {
      return { error: "Unauthorized" };
    }
    ```
  </Accordion>
  
  <Accordion title="Use ADMIN for Full Module Access">
    The `ADMIN` action grants full access to a module, so check for either specific actions or ADMIN:
    
    ```typescript
    if (ability.can("delete", "users") || ability.can("manage", "users")) {
      // User can delete
    }
    ```
  </Accordion>
  
  <Accordion title="Combine with Organization Isolation">
    Always combine permission checks with organization filtering:
    
    ```typescript
    if (!ability.can("read", "forest-patrimony")) {
      return { error: "Unauthorized" };
    }
    
    const data = await prisma.forestPatrimony.findMany({
      where: { organizationId: session.user.organizationId }
    });
    ```
  </Accordion>
  
  <Accordion title="Refresh Permissions After Role Changes">
    When updating user roles, ensure the user re-authenticates or refresh their session to get updated permissions.
  </Accordion>
</AccordionGroup>

## Client-Side Permission Checks

For UI conditional rendering:

```typescript
"use client";

import { useSession } from "next-auth/react";

export function ActionButton() {
  const { data: session } = useSession();
  const permissions = session?.user?.permissions ?? [];
  
  const canCreate = permissions.includes("users:CREATE") || 
                    permissions.includes("users:ADMIN");
  
  if (!canCreate) return null;
  
  return <button>Create User</button>;
}
```

<Warning>
  Client-side permission checks are for UX only. Always enforce permissions on the server.
</Warning>
