# Audit System

The Audit System provides comprehensive logging of all user actions and system events across SMyEG. Every significant operation is tracked with full context for compliance, security, and troubleshooting.

## Audit Log Structure

Each audit log entry captures detailed information about an action:

```typescript
type AuditLog = {
  id: string;
  organizationId?: string;              // Organization context
  userId?: string;                      // User who performed the action
  sessionId?: string;                   // Session identifier
  
  // Action Details
  action: AuditAction;                  // Type of action performed
  entityType?: string;                  // Entity affected (e.g., "User", "Level4")
  entityId?: string;                    // ID of affected entity
  entityName?: string;                  // Name/description of entity
  
  // Change Tracking
  oldValues?: any;                      // Previous values (JSON)
  newValues?: any;                      // New values (JSON)
  changedFields: string[];              // List of changed field names
  
  // Request Context
  ipAddress?: string;                   // Client IP address
  userAgent?: string;                   // Browser/client info
  requestId?: string;                   // Request correlation ID
  durationMs?: number;                  // Operation duration
  statusCode?: number;                  // HTTP status code
  errorMessage?: string;                // Error details if failed
  
  // Additional Context
  metadata?: any;                       // Extra contextual data (JSON)
  createdAt: Date;                      // When action occurred
};
```

## Audit Actions

The system tracks these action types:

```typescript
enum AuditAction {
  CREATE              // Entity created
  UPDATE              // Entity updated
  DELETE              // Entity deleted
  LOGIN               // User logged in
  LOGOUT              // User logged out
  LOGIN_FAILED        // Failed login attempt
  PASSWORD_RESET      // Password reset
  PERMISSION_CHANGE   // Permission modified
  EXPORT              // Data exported
  VIEW                // Entity viewed (sensitive data)
}
```

<Accordion title="Action Details">
  **CREATE** - Records:
  - New entity type and ID
  - All initial field values in `newValues`
  - User who created it
  - Organization context

  **UPDATE** - Records:
  - Entity type and ID
  - Previous values in `oldValues`
  - New values in `newValues`
  - List of changed fields in `changedFields`
  - User who made the update

  **DELETE** - Records:
  - Deleted entity type and ID
  - Final values before deletion in `oldValues`
  - User who deleted it

  **LOGIN/LOGOUT** - Records:
  - User ID
  - IP address
  - User agent
  - Session ID
  - Timestamp

  **LOGIN_FAILED** - Records:
  - Attempted username/email
  - IP address
  - Failure reason (in `errorMessage`)
  - Timestamp

  **EXPORT** - Records:
  - Entity type exported
  - Export format (CSV, XLSX, JSON)
  - Number of records
  - User who exported

  **VIEW** - Records:
  - Sensitive entity viewed
  - User who viewed it
  - Purpose (if required)
</Accordion>

## Partitioning Strategy

Audit logs use **time-based partitioning** for performance:

```sql
-- Partitioned by month for efficient queries and archival
CREATE TABLE audit_logs_2024_03 PARTITION OF audit_logs
  FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');

CREATE TABLE audit_logs_2024_04 PARTITION OF audit_logs
  FOR VALUES FROM ('2024-04-01') TO ('2024-05-01');
```

<Note>
Partitioning improves query performance and simplifies archival. Old partitions can be archived or dropped without affecting current data.
</Note>

## Indexing

Key indexes for fast audit log queries:

```sql
-- Fast lookups by user
CREATE INDEX idx_audit_logs_user_id ON audit_logs(userId);

-- Fast lookups by organization
CREATE INDEX idx_audit_logs_organization_id ON audit_logs(organizationId);

-- Fast lookups by action and time
CREATE INDEX idx_audit_logs_action_created_at ON audit_logs(action, createdAt DESC);

-- Fast lookups by entity
CREATE INDEX idx_audit_logs_entity ON audit_logs(entityType, entityId);

-- Fast lookups by session
CREATE INDEX idx_audit_logs_session_id ON audit_logs(sessionId);
```

## Automatic Audit Logging

Audit logs are created automatically by:

### API Middleware
All API routes automatically log:
- Request method and path
- User and organization context
- Request duration
- HTTP status code
- Errors and stack traces

### Database Triggers
Database-level triggers can log:
- Direct database modifications
- Schema changes
- Data imports

### Application Layer
Explicit audit calls in critical operations:

```typescript
import { createAuditLog } from '@/lib/audit';

// Log a critical action
await createAuditLog({
  userId: session.user.id,
  organizationId: session.user.organizationId,
  action: 'UPDATE',
  entityType: 'Level4',
  entityId: level4.id,
  entityName: level4.name,
  oldValues: { status: 'DRAFT' },
  newValues: { status: 'ACTIVE' },
  changedFields: ['status'],
  metadata: { reason: 'Approved by manager' }
});
```

## Security and Compliance

### Immutability
Audit logs should be **write-only** in production:
- No user updates allowed
- Deletions restricted to admins only
- Consider using append-only storage

### Retention Policies
Define retention based on compliance requirements:
- **Financial records**: 7+ years
- **Security events**: 1-3 years
- **General operations**: 6-12 months

### Access Control
Restrict audit log access:
- Read: Auditors, security team, managers
- Export: Compliance officers
- Delete: System administrators only

### Sensitive Data
Be careful with PII in audit logs:
- Hash or mask sensitive fields
- Don't log passwords or tokens
- Redact financial details if not needed

## Code Examples

### Querying Audit Logs

```typescript
// Fetch recent login attempts
const response = await fetch(
  '/api/audit?action=LOGIN&page=1&limit=50'
);
const result = await response.json();
const loginLogs = result.data.items;

// Fetch all changes to a specific entity
const entityResponse = await fetch(
  '/api/audit?entityType=Level4&page=1&limit=100'
);
const entityAudit = await entityResponse.json();

// Analyze failed login attempts
const failedLogins = await fetch(
  '/api/audit?action=LOGIN_FAILED&page=1&limit=100'
);
const failures = await failedLogins.json();
console.log(`${failures.data.total} failed login attempts`);
```

### Exporting Audit Logs

```typescript
// Export as JSON
const jsonExport = await fetch('/api/audit', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ format: 'json' })
});
const jsonData = await jsonExport.json();

// Export as CSV
const csvExport = await fetch('/api/audit', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ format: 'csv' })
});
const csvData = await csvExport.json();
const csvContent = csvData.data.csv;

// Download CSV
const blob = new Blob([csvContent], { type: 'text/csv' });
const url = URL.createObjectURL(blob);
const a = document.createElement('a');
a.href = url;
a.download = `audit-logs-${Date.now()}.csv`;
a.click();
```

## Best Practices

1. **Regular monitoring** - Review audit logs regularly for suspicious activity
2. **Partition management** - Archive old partitions quarterly/annually
3. **Index maintenance** - Rebuild indexes on active partitions monthly
4. **Sensitive data** - Never log passwords, tokens, or credit card numbers
5. **Performance** - Use specific filters (action, entityType) to avoid full table scans
6. **Compliance** - Export logs regularly for external audits
7. **Testing** - Test audit logging in your CI/CD pipeline
8. **Documentation** - Document custom audit actions for your team
